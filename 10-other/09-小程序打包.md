# 小程序打包

小程序打包是将开发代码编译成可在微信客户端运行的包的过程。微信对小程序包大小有严格限制，合理规划分包是小程序性能优化的关键环节。

## 包大小限制

目前小程序分包大小有以下限制：

- 整个小程序所有分包大小不超过 **20M**（开通虚拟支付后的小游戏不超过 30M）
- 单个分包/主包大小不能超过 **2M**

超过限制后无法上传和发布，因此大型项目必须进行分包处理。

## 分包处理

### 基本概念

在构建小程序分包项目时，会输出一个或多个分包。每个使用分包的小程序必定含有一个**主包**。

- **主包**：放置默认启动页面 / TabBar 页面，以及所有分包都需用到的公共资源 / JS 脚本
- **分包**：根据开发者配置进行划分，按需加载

### 打包原则

- 声明 `subpackages` 后，将按 `subpackages` 配置路径进行打包，配置路径外的目录将被打包到主包中
- 主包也可以有自己的 pages（即最外层的 pages 字段）
- `subpackage` 的根目录不能是另外一个 `subpackage` 内的子目录
- `tabBar` 页面必须在主包内

### 配置方式

在 `app.json` 中通过 `subpackages` 字段声明分包结构：

```json
{
  "pages": [
    "pages/index/index",
    "pages/logs/logs"
  ],
  "subpackages": [
    {
      "root": "packageA",
      "name": "packA",
      "pages": [
        "pages/cat/cat",
        "pages/dog/dog"
      ]
    },
    {
      "root": "packageB",
      "name": "packB",
      "pages": [
        "pages/apple/apple",
        "pages/banana/banana"
      ]
    }
  ]
}
```

### 分包路由跳转

分包页面之间的跳转和普通页面一样，使用 `wx.navigateTo` 等 API，路径带上分包 root 前缀即可：

```js
// 从主包跳转到分包页面
wx.navigateTo({
  url: '/packageA/pages/cat/cat'
})

// 分包之间跳转
wx.navigateTo({
  url: '/packageB/pages/apple/apple'
})
```

## 依赖打包规则

分包中引用的依赖和资源，遵循以下规则决定最终打入哪个包：

### 主包和分包同时使用了一个依赖

主包不能引用分包资源，但分包可以引用主包资源。为了让两个包都能引用到，**该依赖只能打入主包**。

### 某个依赖只在分包中使用

**打入当前分包**，不会影响主包体积。

### 两个分包使用了同一个依赖或资源

该依赖/资源会被**打入主包**，避免重复打包。

```
依赖 A（仅主包使用）     → 主包
依赖 B（仅分包A使用）    → 分包A
依赖 C（主包+分包A使用） → 主包
依赖 D（分包A+分包B使用） → 主包
```

### 兼容处理

微信后台会编译两份代码包：

- **分包后代码**：新客户端使用，按分包加载
- **整包兼容代码**：老客户端使用，把各个 `subpackage` 里面的路径放到 pages 中

所以使用分包无需担心低版本兼容问题。

## 独立分包

### 什么是独立分包

普通分包需要依赖主包才能运行，而独立分包可以**独立于主包运行**。设置为独立分包后，小程序启动时只加载该分包即可打开页面，不需要先下载主包。

- 普通分包启动顺序：**加载主包 → 再加载当前分包**
- 独立分包启动顺序：**直接加载当前分包**

适合作为小程序入口页面的分包，加载速度更快，用户体验更好。

### 配置方式

和普通分包配置一样，加上 `independent: true` 即可：

```json
{
  "subpackages": [
    {
      "root": "packageA",
      "pages": [
        "pages/cat/cat"
      ],
      "independent": true
    }
  ]
}
```

### 独立分包的限制

不能把所有分包都设为独立分包，原因如下：

1. **无法可靠获取全局状态**：独立分包不依赖主包，不一定能获取到 `getApp()`。只有在主包被加载后，独立分包才能访问到全局 App 实例，属于**概率性问题**，调试困难
2. **不支持使用插件**：独立分包中无法使用 `plugins` 声明的插件
3. **公共样式不生效**：小程序的 `app.wxss` 或 Taro 的 `app.less` 等全局样式在独立分包中不生效

::: warning 注意
独立分包中不能使用 `getApp()` 获取全局状态，如果需要共享数据，应使用 wx.setStorageSync 或其他独立于主包的通信方式。
:::

## 分包预下载

### 为什么需要预下载

分包的加载时机默认是**进入分包页面时才开始下载**。这导致首次跳转分包页面时可能出现短暂白屏：

> 白屏耗时 = 下载分包时间 + 接口请求时间 + 视图渲染时间

用户体验差，尤其在弱网环境下更为明显。

### preloadRule 配置

在 `app.json` 中配置 `preloadRule`，可以在特定场景下**预先下载分包资源**，用户真正跳转时可以直接使用已下载的分包，消除等待：

```json
{
  "preloadRule": {
    "pages/index/index": {
      "network": "all",
      "packages": ["packageA"]
    },
    "packageA/pages/cat/cat": {
      "network": "wifi",
      "packages": ["packageB"]
    }
  }
}
```

- `network`：指定预下载的网络条件，`all`（不限制）或 `wifi`（仅 WiFi 下预下载）
- `packages`：指定要预下载的分包 root 或 name

### 使用建议

- 在**首页或高流量页面**配置预下载规则，提前加载用户可能访问的分包
- 移动网络下（非 WiFi）谨慎预下载大体积分包，避免消耗用户流量
- 同一个分包可以被多个页面配置预下载，已下载的分包不会重复下载

## 打包优化建议

1. **图片等静态资源放 CDN**，不要在本地打包 — 腾讯云、阿里云 OSS 等都有免费额度
2. **公共组件和工具函数放入主包**，避免分包间重复引用导致打入主包后体积不受控
3. **按业务模块划分分包**，控制单个分包大小在 2M 以内
4. **使用代码依赖分析工具**定期检查包体积分布
5. **Taro/uniapp 等框架**也支持分包配置，配置方式与原生基本一致，在对应的 `app.config.js` 或 `pages.json` 中设置即可
6. **启用上传压缩**：开发者工具上传时会自动对代码进行压缩混淆，确保工具版本为最新以获取最佳的压缩效果
